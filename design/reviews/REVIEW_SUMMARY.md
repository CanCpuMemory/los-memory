## 四、关键决策建议

### 4.1 需团队决策的问题

#### 决策1：Knowledge 最终归属

**选项A**: 归为扩展层（架构师建议）
- 理由：数据模型与 Observation 不同，使用场景分离
- 风险：若后续证明高频联动，升级成本较高

**选项B**: 设为核心候选，6个月后评估
- 理由：概念邻近，可观察实际使用模式
- 风险：期间需维护"可能升级"的兼容性

**建议**: 选择 **选项A（扩展层）**，但提供平滑升级路径

#### 决策2：Approval 迁出时间表

**原方案**: 6个月迁出
**建议调整**: 12个月分三阶段

| 阶段 | 时间 | 动作 |
|------|------|------|
| 阶段1 | 1-3月 | 冻结功能，添加废弃警告 |
| 阶段2 | 4-8月 | VPS Agent Web 并行构建 |
| 阶段3 | 9-12月 | 数据迁移，双系统运行 |

**建议**: 采纳调整方案，设立跨项目协调委员会

#### 决策3：扩展默认状态

**选项A**: 默认启用现有扩展（保持向后兼容）
**选项B**: 默认禁用所有扩展（明确分层）

**建议**: **选项A（默认启用现有扩展）**，但添加废弃警告，新用户使用选项B

### 4.2 默认决策（无需会议）

## 六、各角色详细评审意见

### 6.1 系统架构师评审

**总体评价**: 方案架构合理，但需要关键调整

**优势**:
- 两步判定框架（成熟度+归属）避免主观判断
- 三层架构（core/extensions/migrate_out）关注点分离清晰
- 兼容性承诺分级保护核心用户

**主要关切**:
1. Knowledge 判定条件不充分，需要明确决策
2. Approval 6个月迁出过于乐观，建议12个月
3. 动态扩展加载增加运行时不确定性，建议静态化

**关键建议**:
- 增加 "core-extension" 中间层表达 Knowledge 的过渡状态
- 建立跨项目协调委员会监督 Approval 迁出
- 收集 Knowledge 使用数据，6个月后评估升级

### 6.2 后端工程师评审

**总体评价**: 技术可行，但需谨慎处理三个技术难点

**代码规模分析**:
- 44个模块，15,788行代码
- 依赖关系复杂，需要仔细解耦

**三大技术难点**:
1. **CLI 动态注册**: 需要保证注册顺序，避免循环依赖
2. **数据库事务边界**: 连接池和事务传播需要测试覆盖
3. **Approval 数据迁移**: 多表关联，建议双写方案

**推荐工具**:
- `pydeps`: 依赖关系可视化
- `yoyo-migrate`: 数据库迁移
- `dependency-injector`: 依赖注入解耦

**实施建议**:
- 8周分阶段实施，每阶段有明确验收标准
- 每阶段后运行回归测试套件
- 保留 feature/architecture-convergence 分支

### 6.3 CLI/工具专家评审

**总体评价**: UX 良好，但需要增强扩展管理能力

**当前问题**:
- 扩展命令与核心命令在 help 中无区分
- 用户无法控制扩展加载
- 扩展加载失败导致 CLI 崩溃

**优先改进项**:
1. 添加 `admin extensions list/status/enable/disable` 命令
2. 实现懒加载，100ms 性能预算
3. Help 输出添加 `[EXT]` 标记和实验性警告

**代码示例**（改进后 help 输出）:
```
Core Commands:
  memory        Memory data access commands
  observation   Observation management commands

Extension Commands:
  incident [E]  Incident management (load: 15ms)
  recovery [E]  Recovery management (load: 22ms)
  approval [X]  Approval workflow [EXPERIMENTAL] [DEPRECATED]
```

### 6.4 技术文档工程师评审

**总体评价**: 文档与实现严重不一致，需立即更新

**主要问题**:
- 5项实际能力（incident/recovery/approval/knowledge/attribution）未在架构文档中声明
- 版本号混乱（v0.2.0 vs 1.0.0 vs v1.2）
